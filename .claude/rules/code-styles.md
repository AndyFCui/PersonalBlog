# Code Styles

## 目标
让代码短、直、稳、可维护，而不是花哨。

## 通用风格

- 优先使用清晰命名，不使用炫技缩写
- 单函数单职责
- 优先早返回，减少嵌套
- 超过三层缩进，优先重构
- 避免隐式副作用
- 避免横向过宽的逻辑块
- 避免“聪明代码”，优先“清楚代码”

## 分支控制

- 优先消除特殊情况，不优先添加特殊分支
- 分支过多时，先检查数据结构是不是错了
- 对同类条件优先用表驱动/映射，而不是 if/else 链

## 抽象原则

- 只有在重复真实出现后再抽象
- 不为假想复用创建抽象层
- 抽象必须降低调用方复杂度，否则视为失败抽象

## 命名

- 变量名说清角色，不说模糊感觉
- 函数名优先动词开头
- 布尔变量优先 is/has/can/should
- 避免 tmp、data、info、handler 这类泛词滥用

## 注释规范

- 注释写“为什么”，不是机械复述“做了什么”
- 块注释用于说明模块意图、边界、约束
- TODO 必须带上下文和下一步动作

示例：

```ts
/* ------------------------- 失败重试：只覆盖幂等请求 ------------------------- */
/* 原因：
 * 非幂等请求重复执行可能带来副作用，因此这里只对 GET / 可重试操作做统一重试。
 */
```

## 代码结构

推荐顺序：

1. type / interface
2. 常量
3. 主流程函数
4. 子函数
5. 导出

## 禁止项

- 超长函数
- 深层嵌套
- 魔法数字散落
- 重复逻辑复制粘贴
- 无意义包装函数
- 不必要的类
- 伪泛化抽象
